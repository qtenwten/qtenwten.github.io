## 1. How SEO Metadata Is Formed

### 1.1 Title, Description, Keywords

**Source of truth**: `src/config/routeSeo.js` â€” static `ROUTE_SEO` object (~300 lines) with per-route, per-language SEO config.

**Runtime**: `SEO.jsx` component receives `title`, `description`, `path` props â†’ calls `getRouteSeo(language, cleanPath)` â†’ uses `routeSeo.title` / `routeSeo.description` as defaults if props not provided.

**Build-time**: `generate-pages.js` reads from `routeSeo.js` via `getAllLocalizedSeoPages()` â†’ injects into pre-rendered HTML.

**Article pages**: SEO data comes from article fields (`seoTitle`, `seoDescription`) returned by Worker API, merged with `routeSeo.js` fallbacks.

**Issue**: `routeSeo.js` has hardcoded static data â€” not derived from `routeRegistry.js` titleKey/descriptionKey + locales. When you change a title in `locales/ru.json`, you must also update `routeSeo.js` manually.

### 1.2 Canonical URL

**Runtime**: `SEO.jsx` generates `canonicalUrl` via `getLocalizedRouteUrl(language, cleanPath)` â†’ always points to the current language/version.

**Article pages**: `canonicalUrl` is `getLocalizedRouteUrl(language, '/articles/${slug}')` â€” language-specific canonical.

**Pre-rendered**: `generate-pages.js` injects `<link rel="canonical" href="${page.url}" />` where `page.url` = `getLocalizedRouteUrl(language, path)`.

**Correct**: Canonical is always language-specific, pointing to the correct localized URL. âś…

### 1.3 Hreflang

**Runtime**: `SEO.jsx` always outputs hreflang for BOTH ru and en, regardless of whether the page has both language versions:
```jsx
<link rel="alternate" hreflang="ru" href={ruUrl} />
<link rel="alternate" hreflang="en" href={enUrl} />
<link rel="alternate" hreflang="x-default" href={ruUrl} />
```

**Article pages**: `ArticlePage.jsx` conditionally outputs hreflang based on whether translated slugs exist for each language. âś…

**Pre-rendered (tool pages)**: `generate-pages.js` `buildAlternateLinks()` always outputs both ru/en hreflang. âś…

**Sitemap**: `buildSitemap()` in `generate-pages.js` generates correct hreflang with `x-default` pointing to RU.

**Issue**: For pages that only exist in RU (or don't have translations), `SEO.jsx` still outputs both ru/en hreflang pointing to the same content. This is technically acceptable (Google handles it), but not ideal.

### 1.4 Robots

| Page | Robots Value | Source |
|------|-------------|--------|
| Most pages | `index,follow` | `routeSeo.js` default |
| `/search` | `noindex,follow` | `SEO.jsx` prop + `routeSeo.js` |
| `/articles` | `noindex,follow` | `routeSeo.js` |
| 404 | `noindex,nofollow` | `NotFound.jsx` |
| Article pages | `index,follow` | Default from `SEO.jsx` |

**Pre-rendered**: `generate-pages.js` injects `robots` from page config, plus `googlebot` and `yandex` copies. âś…

**Issue**: Robots meta is not set on pre-rendered 404 pages â€” `generate-pages.js` doesn't prerender NotFound.

### 1.5 Sitemap

**Generated by**: `generate-pages.js` `buildSitemap()` function

**Logic**:
- Pages with `includeInSitemap: false` are excluded
- Article pages get hreflang for only languages that exist
- Non-article pages get both ru/en hreflang

**Included**: All tool pages (RU + EN), home pages (RU + EN), articles (only languages with actual articles)

**Excluded**: `/search`, `/articles` (index), 404

**lastmod**: Always `new Date().toISOString().slice(0, 10)` â€” today's date, not the actual content modification date.

**Issue**: `lastmod` is always today's date â€” not the article's `publishedAt` date. Search engines can't know if content actually changed.

### 1.6 Structured Data

**Tool pages**: `SEO.jsx` outputs default `WebSite` schema:
```json
{
  "@context": "https://schema.org",
  "@type": "WebSite",
  "potentialAction": {
    "@type": "SearchAction",
    "target": "https://qsen.ru/{language}/?search={search_term_string}"
  }
}
```

**Article pages**: `ArticlePage.jsx` outputs `Article` schema:
```json
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "...",
  "description": "...",
  "author": { "@type": "Person", "name": "..." },
  "datePublished": "...",
  "image": [...],
  "publisher": { "@type": "Organization", "name": "QSEN.RU", ... }
}
```

**Missing from Article schema**: `articleSection`, `wordCount`, `timeToRead` â€” common Article structured data fields.

**At runtime**: `SEO.jsx` â†’ `getRouteSeo()` â†’ same `routeSeo.js` data

### 3.2 Bilingual Architecture

- `/ru/*` and `/en/*` routes are separate routes in React Router
- `SEO.jsx` generates language-specific canonical and hreflang based on current `language` context
- Pre-rendered pages are generated for both locales separately
- Article multilingual switching uses `translationKey` to match RU/EN versions

### 3.3 Article SEO Specifics

```
Worker API returns article with:
  - slug (may differ by language)
  - language (ru/en)
  - translation_key (shared across translations)
  - seoTitle, seoDescription (may differ by language)
  - publishedAt, author, coverImage
```

Pre-rendered article pages use these fields directly. Runtime `ArticlePage.jsx` uses `seoTitle` / `seoDescription` with fallbacks.

## 4. SEO Risks

### 4.1 High Risk

**Risk 1: `lastmod` in sitemap is always today's date**
- `sitemap.xml` has `<lastmod>2026-04-22</lastmod>` for ALL pages
- Article `publishedAt` is not used
- Search engines can't prioritize crawling based on actual content changes
- **Impact**: Wasted crawl budget, stale content not prioritized

**Risk 2: Pre-rendered pages may not match runtime SEO for articles**
- Pre-rendered article HTML uses SEO data fetched at build time
- If Worker API returns different data at build vs runtime, there's a mismatch
- No validation that pre-rendered SEO matches runtime
- **Impact**: SEO divergence between crawler (sees pre-rendered) and user (sees runtime)

**Risk 3: `/articles/:slug` (no lang prefix) legacy redirect**
- `/articles/test-slug` â†’ `/ru/articles/test-slug`
- If the article is English-only, user lands on RU article list (wrong language)
- **Impact**: English users redirected to Russian content

### 4.2 Medium Risk

**Risk 4: `routeSeo.js` is manually maintained static data**
- 300 lines of hardcoded titles/descriptions in two languages
- Easy to forget to update when changing locale strings
- No build-time validation that all routes have SEO entries
- **Impact**: Missing or outdated SEO data

**Risk 5: `SEO.jsx` outputs hreflang for both languages even on untranslated pages**
- Tool page without EN version still gets `hreflang="en"` pointing to same URL
- Technically accepted by Google but not clean
- **Impact**: Minor â€” Google is tolerant of this

**Risk 6: No `article:published_time` or `article:modified_time` in Article schema**
- Article structured data lacks common fields like `dateModified`, `articleSection`, `wordCount`
- **Impact**: Missed rich result opportunities

**Risk 7: No breadcrumb structured data on article pages**
- `BreadcrumbsContext` tracks breadcrumb trail
- But no `BreadcrumbList` schema is rendered
- **Impact**: Missed breadcrumb rich results

**Risk 8: Missing og:image validation**
- All pages use the same `https://qsen.ru/og-image.png` fallback
- Article pages use `article.coverImage` if available
- No validation that og:image is correct dimensions (1200x630)
- **Impact**: Suboptimal social sharing

### 4.3 Low Risk

**Risk 9: No `keywords` meta tag on most pages**
- `routeSeo.js` has `keywords` for tool pages but `SEO.jsx` doesn't render them
- `MetaTagsGenerator` page has its own keywords input but they're not rendered on the page itself
- **Impact**: Minor â€” Google ignores `keywords` meta for ranking

**Risk 10: `/qr-code-generator` is client-only â€” no pre-rendered SEO shell**
- First crawl sees minimal content (no pre-render)
- **Impact**: QR Code Generator is harder for search engines to index properly

**Risk 11: No `lang` attribute on `<html>` at build time**
- `generate-pages.js` replaces `<html lang="...">` âś…
- But if JavaScript disabled, `<html lang>` is set
- `SEO.jsx` sets `lang` via `<html lang={language} />` âś…
- **Impact**: OK for modern crawlers

**Risk 12: `MetaTagsGenerator` page itself has no special SEO**
- Tool page uses standard tool page SEO
- But it's a tool that generates meta tags for OTHER pages â€” ironic
- **Impact**: None

## 5. Internal Linking Analysis

### 5.1 Current Internal Links

| Source | Target | Method |
|--------|--------|--------|
| Home tool cards | Tool pages | `<Link>` |
| RelatedTools | Other tool pages | `<Link>` + preloadRoute |
| Header logo | Home | `<Link>` |
| Header Articles | Articles index | `<Link>` |
| Article tool badge | Linked tool | `<Link>` |
| Breadcrumbs | Parent pages | `<Link>` |

### 5.2 Weaknesses

**Weak link equity distribution**:
- Tool pages have no links FROM other pages except home and relatedtools
- "SEO Audit Pro" has `priority 0.9` in sitemap but isn't prominently linked from home
- Articles could link more to related tools

**No cross-linking between related tools**:
- `RelatedTools` shows ALL other tools, not semantically related ones
- No tool-to-tool article connections

**Article internal links**:
- Article markdown content links to tool pages via `RelatedTools` at bottom
- But articles don't embed links to other articles
- Articles list has no "related articles" in the list view (only detail page has sidebar)

**No breadcrumbs on tool pages** (only on article pages):
- `BreadcrumbsContext` exists but `ToolPageShell` doesn't render breadcrumbs
- Only article pages use breadcrumbs

**No sitelinks** (Google search results sub-links):
- Home page doesn't have structured navigation for main sections
- Could add `SiteNavigationElement` schema

## 6. Recommendations

### P0 â€” Critical

**P0-1: Fix sitemap `lastmod` to use actual dates**
- Use article `publishedAt` for article pages
- Use build date only for static pages
- Effort: 30 min
- Impact: Search engines prioritize fresh content correctly

**P0-2: Fix article legacy redirect language handling**
- `/articles/:slug` should check article language before redirecting
- If article is EN-only, redirect to `/en/articles/:slug`, not `/ru/articles/:slug`
- Effort: 1 hour
- Impact: English users see correct content

### P1 â€” High

**P1-1: Add Article structured data fields**
- Add `dateModified`, `articleSection`, `wordCount` to Article schema
- Consider `BreadcrumbList` for article pages
- Effort: 2 hours
- Impact: Rich result opportunities

**P1-2: Validate og:image dimensions**
- Check that `article.coverImage` meets 1200x630 requirement
- Add warning if image is too small
- Add `og:image:width` / `og:image:height` dynamically
- Effort: 1 hour
- Impact: Better social sharing

**P1-3: Add breadcrumb structured data**
- Render `BreadcrumbList` schema on article pages
- `BreadcrumbsContext` already tracks the trail
- Effort: 1 hour
- Impact: Breadcrumb rich results in Google

**P1-4: Derive `routeSeo.js` from `routeRegistry.js` + locales**
- Eliminates manual duplication of title/description
- Changes to locale strings automatically reflected in SEO
- Effort: 4-6 hours
- Impact: Single source of truth, fewer SEO bugs

### P2 â€” Medium

**P2-1: Add SiteNavigationElement schema to home page**
- Helps Google understand site structure
- Enables sitelinks in search results
- Effort: 1 hour
- Impact: Better search appearance

**P2-2: Pre-render QRCodeGenerator SEO shell**
- Currently fully client-rendered
- Search engines see minimal content
- Effort: 2 hours
- Impact: Better indexing of QR pages

**P2-3: Add semantic section links in articles**
- Connect related articles via in-content links
- Article list could show "related articles" before detail page
- Effort: 4 hours
- Impact: Better crawl depth

**P2-4: Reduce hreflang noise**
- Only output hreflang for languages that actually exist for that content
- Currently outputs ru/en for all tool pages regardless of translation
- Effort: 2 hours
- Impact: Cleaner hreflang signals
